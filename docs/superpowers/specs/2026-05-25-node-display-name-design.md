# 节点自定义命名设计

日期：2026-05-25

## 背景

当前方案中心和工作台中的节点主要通过默认序号文案展示，例如“节点 1”。这个文案在多个链路中被直接拼接使用，包括：

- 方案中心节点头部
- 工作台节点头部
- 自定义字段影响范围弹窗
- 节点复制后的绑定提示文案

这导致用户无法给节点表达业务语义，也无法在跨节点绑定和后续使用时快速识别节点。

## 目标

支持为每个节点设置可持久化的自定义名称，并让所有当前依赖“节点 N”显示的链路统一改为优先显示用户命名。

## 非目标

- 不改变节点 `id` 的生成规则
- 不把节点命名抽成独立映射表
- 不影响现有组件类型 `packageType` 的业务含义

## 数据模型

在方案节点持久化结构中新增字段：

- `displayName?: string`

规则：

- 为空字符串或缺失时，节点视为“未命名”
- 未命名节点回退显示默认文案 `节点 N`
- 已命名节点统一显示 `displayName`

## 显示规则

新增统一的节点显示名获取逻辑，所有相关 UI 不再自行拼接 `节点 ${index + 1}`。

统一规则：

1. 若节点存在非空 `displayName`，显示该名称
2. 否则显示 `节点 N`

## 影响范围

需要统一替换以下链路：

- 方案中心节点卡片头部的“节点 N”
- 工作台节点卡片头部的“节点 N”
- 自定义字段编辑弹窗中的节点标签
- 复制节点时提示文案中的目标节点名称
- 所有当前依赖节点序号名称的辅助展示

## 编辑方式

在方案中心节点头部提供节点命名入口，使用户可以直接修改当前节点的 `displayName`。

交互要求：

- 仅草稿状态允许编辑
- 发布态继续只读
- 编辑后走现有草稿保存链路持久化

## 复制规则

复制节点后，新节点的 `displayName` 默认清空。

原因：

- 避免多个节点共享同名，造成识别混乱
- 复制后若用户需要命名，应按新节点语义重新设置

## 技术实现方向

- 节点序列化持久化时写入 `displayName`
- 节点 hydrate/runtime 创建时保留 `displayName`
- 增加统一工具函数生成节点显示名
- 所有相关组件改为调用统一函数

## 验证范围

需要验证：

- 草稿节点命名后，保存不丢失
- 已命名节点在方案中心与工作台显示一致
- 自定义字段绑定弹窗中显示命名后的节点名
- 节点复制后名称按规则回退为空，并显示新的默认序号名
- 已发布方案加载后仍能显示持久化的节点名称
