# 达摩盘自动化圈人插件设计

## 目标

为当前项目中的“自动化圈人”按钮提供一条真正可交付给用户使用的自动化链路。

期望体验是：

1. 用户在 Chrome 中安装一次插件。
2. 用户继续像现在一样使用当前项目页面生成 JSON。
3. 用户点击“自动化圈人”。
4. Chrome 自动打开或切换到达摩盘圈人页面。
5. 插件自动完成固定的 3 步操作：
   - 点击“参数粘贴”
   - 将当前 JSON 写入弹窗输入框
   - 点击“确定”

## 为什么改用插件而不是 Python 浏览器自动化

当前的 Python + Chrome DevTools 方案更适合本地调试，不适合最终用户，原因有：

- 依赖浏览器通过调试端口启动
- 依赖本地 Python 环境
- 依赖接管浏览器进程
- 对普通用户安装和使用门槛过高

Chrome 插件更适合作为正式方案，因为它：

- 运行在用户自己的正常 Chrome 会话里
- 可以直接操作当前浏览器中的达摩盘页面
- 不要求用户改 Chrome 启动方式
- 更接近“安装一次，之后一直可用”的产品形态

## 范围

### 本期范围

- 保留当前项目页中的“自动化圈人”按钮
- 提供一个 Chrome 插件，内部通过开发者模式安装
- 当前开发阶段支持本地域名 `http://127.0.0.1:5173`
- 后续上线时支持正式部署域名
- 自动化行为仅限达摩盘圈人页面
- 自动化行为仅限用户提供的 3 个固定 XPath 动作

### 本期不做

- Chrome 商店发布
- 兼容 Chrome 以外的浏览器
- 通用网页自动化能力
- 批量队列或多标签页并发调度
- 从达摩盘页面回读数据到项目中
- 改造成通用选择器方案，除非后续页面结构发生变化

## 方案对比

### 方案 A：项目页直接给插件发消息

这是推荐方案。

执行流程：

1. 项目页生成当前最终 JSON
2. 项目页向 Chrome 插件发送消息
3. 插件后台校验消息来源域名
4. 插件打开达摩盘标签页，或切换到已有达摩盘标签页
5. 插件把 JSON 转发给达摩盘页面中的 content script
6. content script 在达摩盘页面执行固定 3 步自动化
7. 插件把成功或失败结果回传给项目页

优点：

- 数据流清晰
- 与当前项目页面结合自然
- 本地开发和正式部署都容易兼容
- 不依赖 Python 控制浏览器
- 更容易做来源校验和权限收敛

缺点：

- 需要实现网页与插件之间的消息握手

### 方案 B：通过 URL 参数或浏览器存储中转 JSON

执行流程：

1. 项目页把 JSON 写进 URL 参数或浏览器本地存储
2. 插件或达摩盘页脚本读取 JSON
3. 执行固定 3 步自动化

优点：

- 初版实现较快

缺点：

- 数据链路不够干净
- JSON 可能暴露在地址栏或浏览器存储中
- 后续维护成本更高

不推荐，除非后面消息通信存在明确限制。

### 方案 C：插件只在达摩盘页侧工作

执行流程：

1. 当前项目页只负责打开达摩盘
2. 用户再手动触发插件
3. 插件在达摩盘页执行自动化

优点：

- 实现最简单

缺点：

- 不满足“一键自动化圈人”的目标体验

不推荐。

## 最终方案

采用方案 A：**项目页发消息给插件，插件负责打开达摩盘并执行自动化。**

## 插件结构设计

### 1. Background Service Worker

职责：

- 接收当前项目页发来的消息
- 校验发送方域名是否在允许名单内
- 查找当前浏览器里是否已有达摩盘标签页
- 若没有则新开达摩盘标签页
- 等待达摩盘 content script 就绪
- 把 JSON 发送给达摩盘页脚本
- 收集执行结果并返回给项目页

### 2. 达摩盘 Content Script

只注入到：

- `https://databank.tmall.com/*`

职责：

- 监听插件后台转发过来的自动化命令
- 等待并定位 3 个固定 XPath 元素
- 点击“参数粘贴”
- 向弹窗 textarea 写入 JSON
- 点击“确定”
- 把执行成功或失败结果回传给插件后台

### 3. 插件 Popup

本期不是必须。

可以后续再加，用于：

- 查看插件是否在线
- 显示最近一次自动化状态
- 提供手动重试入口

## 当前项目页改造

当前 `cdp-web/src/components/NormalMode.vue` 中的“自动化圈人”按钮，后续不再调用后端 Python 自动化接口，而是改为调用插件。

新行为应为：

1. 从当前页面状态生成最终 JSON 文本
2. 向插件发送固定格式的消息
3. 插件成功时提示“自动化圈人完成”
4. 插件不存在或未启用时提示用户先安装插件
5. 插件执行失败时展示插件返回的具体错误信息

后端现有的 Python 自动化接口可以暂时保留，用于调试或过渡，但它不再是长期产品方案。

## 消息协议

### 项目页 -> 插件

消息类型：

`CDP_AUTOMATE_DATABANK`

消息结构：

```json
{
  "type": "CDP_AUTOMATE_DATABANK",
  "jsonText": "{ ... 当前生成的人群 JSON ... }",
  "source": "cdp-web"
}
```

### 插件后台 -> 达摩盘 Content Script

消息类型：

`AUTOMATE_DATABANK`

消息结构：

```json
{
  "type": "AUTOMATE_DATABANK",
  "jsonText": "{ ... 当前生成的人群 JSON ... }"
}
```

### 成功返回

```json
{
  "ok": true
}
```

### 失败返回

```json
{
  "ok": false,
  "error": "等待页面元素超时: /html/..."
}
```

## 权限与安全

插件应使用最小权限原则。

建议权限：

- `tabs`
- 如需运行时注入，可加 `scripting`
- Host 权限：
  - `https://databank.tmall.com/*`
  - 当前项目开发地址
  - 后续正式部署地址

安全约束：

- 只接受允许名单中的项目域名消息
- 只在达摩盘域名下运行自动化
- 只执行这 3 个固定 XPath 动作
- 不抓取页面数据
- 不长期保存 JSON
- 一旦来源、URL 或目标元素不符合预期，立即失败

## 用户体验设计

### 正常路径

1. 用户安装插件一次
2. 用户打开当前项目页
3. 用户点击“自动化圈人”
4. 浏览器自动打开或切换到达摩盘页
5. 插件自动执行圈人参数导入
6. 项目页给出成功提示

### 插件未安装或未启用

项目页应提示：

- “未检测到自动化插件，请先在 Chrome 扩展页加载插件。”

### 达摩盘页面执行失败

项目页应提示：

- “自动化圈人失败：未找到参数粘贴入口”

### 页面改版或用户未登录

content script 应返回具体、可判断的问题，而不是泛化报错。

例如：

- “自动化圈人失败：等待参数粘贴入口超时”
- “自动化圈人失败：未检测到目标输入框”

## 错误处理

错误类型建议分成以下几类：

### 1. 插件不可用

场景：

- 页面无法向插件发消息

处理：

- 提示用户安装或启用插件

### 2. 达摩盘标签页打开失败

场景：

- 插件无法新建或定位达摩盘标签页

处理：

- 提示用户重试，或检查浏览器权限

### 3. XPath 元素等待超时

场景：

- 页面未加载完成
- 用户未登录
- 页面结构变更

处理：

- 返回具体 XPath 对应的步骤失败信息

### 4. Content Script 运行异常

场景：

- 页面脚本抛出异常

处理：

- 返回明确异常信息，便于定位

## 测试策略

### 插件后台层

- 测试来源域名校验逻辑
- 测试达摩盘标签页复用或新开逻辑
- 测试消息转发和错误结构

### Content Script 层

- 测试 XPath 等待函数
- 测试点击、输入、点击的固定顺序
- 测试缺失元素时的失败行为

### 集成测试

- 在本地 `http://127.0.0.1:5173` 下联调
- 在达摩盘登录态下手工验证
- 页面结构变化后进行回归验证

## 迁移步骤

1. 在仓库中创建 Chrome 插件骨架，供开发者模式加载
2. 实现达摩盘 content script 的 3 步自动化
3. 实现 background 消息桥和达摩盘标签页管理
4. 将当前项目页“自动化圈人”按钮改成插件消息调用
5. 保留或下线后端 Python 自动化，根据迁移节奏决定
6. 为内部用户补充插件安装说明
7. 正式上线前把正式域名加入插件允许名单

## 已确认决策

- 分发方式：内部使用，开发者模式安装
- 触发方式：项目页触发，达摩盘页执行
- 浏览器形态：使用用户自己的正常 Chrome，而不是独立受控浏览器
- 当前阶段：本地地址先联调，后续切正式域名

## 实施备注

当前的 `cdp_backend.app_factory` 与 `cdp_backend.databank_automation` 中的 Python 自动化实现，不再是长期主路径。它们可以短期保留用于调试，但正式产品路径将转向插件方案。
