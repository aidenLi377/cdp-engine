# DMP 网页与插件采集口径统一设计

日期：2026-07-15

## 背景与目标

当前任务中心可以从网页触发 DMP 采集，但与原 DMP 插件面板存在四处关键差异：网页流程没有使用多条件标签缓存、获取总覆盖人数的时机晚于标签数据计算、Rebase 规则不同、结果字段不完整。因此，同一人群和同一批标签从两个入口启动时，可能得到不同结果。

本次改造目标是让网页和插件共享同一份标签就绪状态与用户设置，并由插件使用原 DMP 插件的请求与计算规则生成最终结果。原有 DataBank 流程、任务记录、方案库和用户隔离逻辑不在改造范围内。

## 核心原则

1. 插件是 DMP 状态、采集和计算的唯一数据来源；网页只负责展示与发起操作。
2. 网页与插件共享 `dmpConditionCache`、字段显示配置和 Rebase 配置。
3. 采集结果始终保存完整字段；字段显示配置只影响界面、复制等显示行为，不删除原始数据。
4. 原始 DMP 插件业务文件不修改。代码改动仅发生在 CDP 项目内置插件及独立合并版本中。
5. 已有任务历史继续兼容旧字段，不迁移或破坏已有数据。

## 方案选择

采用“插件单一数据源”方案：网页通过扩展桥接协议读取和更新共享状态，执行采集时由 DMP 内容脚本读取条件缓存与 Rebase 设置，按照原插件逻辑生成完整结果。

未采用以下方案：

- 网页复制原插件算法：以后修改任意一端都可能再次产生口径漂移。
- 网页操纵原插件面板 DOM：强依赖面板结构，难以稳定自动化，也无法形成清晰的数据协议。

## 共享状态与通信协议

### 共享状态

扩展的 `chrome.storage.local` 保存：

- `dmpConditionCache`：以标签 ID 为键的 `multiGroupOptions` 缓存。
- `columnVisibility`：以标准结果字段为键的显示状态。
- `rebaseExcludedTagIds`：不执行 Rebase 的标签 ID 列表。

标准结果字段顺序与原插件保持一致：

1. 所属大类
2. 标签类型
3. 标签名称
4. 特征明细
5. 人群占比
6. 覆盖人数
7. Rebase
8. Rebase后人数
9. CTR
10. PPC

### 新增协议

网页通过桥接脚本使用两个短请求：

- `CDP_DMP_GET_SETTINGS`：返回条件已就绪标签 ID、字段显示配置和 Rebase 排除列表。
- `CDP_DMP_UPDATE_SETTINGS`：只更新传入的字段显示配置或 Rebase 排除列表，并返回更新后的完整设置。

现有 `CDP_AUTOMATE_DMP_EXTRACT` 保留，但提取阶段不再依赖网页重复传递计算规则。内容脚本从共享存储读取最新条件缓存和 Rebase 配置，避免请求期间两端状态不一致。

桥接响应需要完整透传 `settings`，同时保留现有 `ok`、`error`、`trail`、`crowdId`、`crowdCount` 和 `results` 字段，保证旧消息不受影响。

## 多条件标签行为

1. DMP 页面拦截到带 `multiGroupOptions` 的画像请求后，将其写入 `dmpConditionCache`。
2. 网页打开任务中心或扩展重新连接时读取共享状态。
3. 普通标签始终可以选择。
4. `needCondition: true` 的标签仅在缓存中存在有效的非空条件数组时显示“已就绪”并允许选择；否则显示“待配置”且不能选择。
5. 如果一个已选多条件标签后来失去有效缓存，网页将其从当前选择中移除并提示用户重新配置。
6. 执行标签请求时，普通标签删除基础请求中的 `multiGroupOptions`；多条件标签使用其标签 ID 对应的缓存条件。缺少条件时不发送错误请求，而是返回与原插件一致的“未配置下钻条件”结果行。

## 覆盖人数与 Rebase 计算

插件进入画像透视页并取得有效 API 请求后，先读取页面总覆盖人数，再开始逐标签请求。总覆盖人数沿用原插件的 XPath 和解析规则。

每个结果行：

- `人群占比 = rate × 100 + '%'`
- `覆盖人数 = round(总覆盖人数 × rate)`；无法取得总覆盖人数时为 `-`

随后按照原插件规则，以结果行的“标签名称”为分组键汇总人群占比：

- 标签 ID 在 `rebaseExcludedTagIds` 中：`Rebase` 保持原人群占比，`Rebase后人数` 保持覆盖人数。
- 标签启用 Rebase 且分组总和大于 0：`Rebase = 当前占比 ÷ 分组总和 × 100%`。
- 标签启用 Rebase但分组总和为 0：`Rebase = 0%`。
- `Rebase后人数 = round(总覆盖人数 × Rebase百分比 ÷ 100)`。
- 错误或未配置条件的结果行保持 `-`，不参与汇总。

为保证与原插件直接启动的输出一致，不引入当前网页自动化中 `totalSum > 100.1` 时跳过归一化的额外规则。

## 网页交互设计

### 标签选择区

- 多条件标签保留“多条件”标识。
- 已捕获条件时增加绿色“已就绪”状态并允许勾选。
- 未捕获条件时显示橙色“待配置”，悬停说明需先在 DMP 页面配置该标签。
- 扩展未连接时保持现有整体执行限制，不清空用户已选普通标签。

### 结果工具栏

在现有覆盖人数、行数、复制、导出按钮附近增加：

- “显示字段”：多选标准十字段。修改后立即更新共享存储，表格即时重绘。
- “Rebase”：按标签开关。勾选表示启用 Rebase，取消表示加入排除列表；支持全部启用和全部停用。

控件沿用任务中心现有的浅色卡片、橙色强调色和紧凑排版，不新增全屏步骤或复杂弹窗。

### 表格与历史任务

- 新任务使用标准十字段和共享字段显示配置。
- 当前结果、复制内容和历史结果预览使用当前字段显示配置。
- CSV 导出完整标准字段，保证离线数据不因隐藏列而缺失。
- 历史数据中旧的“点击TGI/转化TGI”继续显示；兼容层将其分别视为 CTR/PPC 的旧别名，不修改数据库原记录。

## 错误处理

- 读取共享设置失败：网页沿用默认字段显示和默认全部 Rebase，同时显示“插件设置同步失败，请刷新后重试”，不阻断普通标签浏览。
- 总覆盖人数读取失败：采集仍可完成，但覆盖人数和 Rebase 后人数显示 `-`，任务结果保留明确提示。
- 条件缓存缺失：该标签不执行远程请求，返回可理解的未配置结果行。
- 设置更新失败：网页回滚本次界面变更，避免出现界面已修改但插件未保存的假状态。
- 扩展旧版本不支持新协议：网页提示需要重新加载新版合并插件，现有 DataBank 功能不受影响。

## 测试与验收

### 自动测试

1. 条件缓存只将有效非空条件标记为已就绪。
2. 多条件请求体使用对应标签的 `multiGroupOptions`。
3. 普通标签请求不会继承其他标签的多条件配置。
4. 给定固定总覆盖人数与接口样本，覆盖人数与原插件结果一致。
5. Rebase 启用、排除、全开和全关结果与原插件公式一致。
6. 标准十字段顺序、旧字段别名兼容和 CSV 完整字段导出正确。
7. 网页设置更新经桥接写入扩展，并能再次读取。
8. 现有插件、前端与后端测试全部通过。

### 手工验收

使用同一 DMP 账号、同一人群包、同一标签组合与同一共享设置，分别从原插件面板和网页任务中心启动采集，对比：

- 多条件标签可选状态一致。
- 总覆盖人数一致。
- 结果行数、字段、标签明细、人群占比和覆盖人数一致。
- Rebase 与 Rebase 后人数一致。
- 网页修改字段或 Rebase 设置后，独立合并插件内的 DMP 面板重新打开能看到相同配置。

## 完成标准

所有自动测试与构建通过；新版合并插件可加载；网页与插件共享设置；固定样本计算一致；原始 DMP 插件业务文件哈希保持不变；现有 DataBank、任务记录和其他网页功能回归通过。
