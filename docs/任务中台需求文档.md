# 任务中台需求文档

## 一、项目目标

希望在现有系统中增加一个“任务中台”能力，让用户不用在官方平台里来回切换和手动操作，而是在我们的系统前端页面中完成整条人群透视链路的发起、查看和结果获取。

用户最终希望达到的效果是：

1. 在我们的系统里输入人群包名称。
2. 在我们的系统里选择需要透视的标签。
3. 点击开始后，系统自动完成后续流程。
4. 用户可以在我们的系统里看到当前执行到哪一步。
5. 最终在我们的系统里拿到透视结果。

## 二、核心价值

任务中台不是简单做一个和官方相似的页面，而是要把原本分散在多个页面、需要人工切换和等待的流程，变成一个可发起、可追踪、可回看的任务流程。

用户使用我们的系统的核心原因包括：

1. 少操作  
   原本需要手工搜索、点击、等待、切换页面，现在希望一键发起。

2. 看得见进度  
   用户希望知道当前任务处于哪个阶段，而不是点完以后没有反馈。

3. 有结果沉淀  
   任务执行完成后，结果应留在我们的系统里，便于复查、下载和后续复用。

4. 前端即可使用  
   用户希望最终部署到服务器后，通过系统前端页面就能完成任务发起和查看。

## 三、用户使用场景

### 场景一：测试数据引擎阶段

用户希望单独测试“数据引擎”这一段流程，而不是一次性执行完整链路。

用途：

1. 验证系统能否打开数据引擎页面。
2. 验证系统能否搜索指定人群包。
3. 验证系统能否精准匹配目标人群包。
4. 验证系统能否点击到应用前的最后一步。

这一阶段的要求是：

1. 只测试到“最终应用按钮之前”。
2. 开发过程中不能误触真实应用动作。
3. 用户需要清楚看到系统已经走到了哪一步。

### 场景二：测试达摩盘阶段

用户希望单独测试“达摩盘”这一段流程。

用途：

1. 验证系统能否打开达摩盘页面。
2. 验证系统能否搜索指定人群包。
3. 验证系统能否精准匹配目标人群包。
4. 验证系统能否读取该人群包对应的信息。
5. 验证系统能否找到“画像透视”入口。
6. 验证系统能否进入透视流程并开始获取数据。

用户希望这一阶段是可见、可跟踪的，而不是点击以后没有反馈。

### 场景三：完整任务流程

最终希望把两段串起来，形成完整任务。

完整任务目标流程为：

1. 数据引擎中定位人群包。
2. 推送到后续平台。
3. 在达摩盘中找到对应人群包。
4. 进入画像透视。
5. 获取所选标签数据。
6. 返回结果到我们的系统。

## 四、任务中台页面需求

任务中台需要作为系统内一个独立入口存在，用户进入后应能完成以下操作。

### 1. 基础输入

页面中需要允许用户输入：

1. 数据引擎测试人群包名称
2. 达摩盘测试人群包名称

后续完整版本中，可以进一步收敛为一个统一的人群包名称输入框，但开发和联调阶段保留两个输入更方便测试。

### 2. 标签选择

页面中需要有“可选标签”区域，用户可以自主选择要透视哪些标签。

第一版只需要支持少量标签测试，当前范围为 3 到 5 个标签，优先保证链路走通。

用户对这个区域的预期是：

1. 能看到标签列表。
2. 能勾选或取消勾选。
3. 系统默认给出一组常用标签。

### 3. 测试按钮

页面中需要有两个明确的测试按钮：

1. 测试数据引擎
2. 测试达摩盘

这两个按钮分别对应两段流程，目的是让用户在开发和联调阶段能分开验证问题，不必每次都跑完整链路。

### 4. 插件状态或本地执行状态

页面中需要展示当前本地执行器是否可用。

用户需要知道：

1. 当前是否已连接。
2. 当前是否可执行。
3. 如果不能执行，是什么原因。

### 5. 任务进度区域

页面中需要有明显的任务进度区域，让用户看到：

1. 当前任务名称。
2. 当前执行阶段。
3. 当前进度百分比。
4. 当前说明文字。

用户要求这部分一定要明确，不要出现“点击后没反应”的情况。

### 6. 任务记录区域

页面中需要有任务记录区，展示历史任务。

用户希望看到：

1. 任务名称。
2. 当前状态。
3. 当前说明。
4. 是否完成。
5. 是否失败。

后续如果任务完成，还应能进一步扩展为查看结果和下载。

## 五、流程要求

### 1. 数据引擎流程要求

用户的要求是：

1. 系统自动打开数据引擎页面。
2. 自动搜索指定人群包名称。
3. 搜索后不能默认使用第一条结果。
4. 必须进行精准匹配。
5. 精准匹配后进入应用动作前的最后一步。
6. 开发阶段停在最终应用按钮之前。

如果未来进入真实执行模式，则在点击应用后，还要继续关注后续状态是否成功。

### 2. 达摩盘流程要求

用户的要求是：

1. 系统自动打开达摩盘页面。
2. 自动搜索指定人群包名称。
3. 搜索后不能默认使用第一条结果。
4. 必须进行精准匹配。
5. 精准匹配后要能读取对应人群包信息。
6. 还要验证是否出现“画像透视”入口。
7. 如果出现该入口，则继续进入透视流程。

### 3. 精准匹配要求

这是用户特别强调的要求。

无论在数据引擎还是达摩盘，搜索结果中的第一条都不一定是目标人群包，因此不能默认使用第一条。

必须做到：

1. 读取搜索结果中的人群包名称。
2. 与用户输入的人群包名称进行精确比对。
3. 只在确认命中目标后，才继续下一步。

### 4. 状态反馈要求

系统必须把任务分阶段反馈给用户，让用户知道当前处于哪一步。

至少应能表达出类似状态：

1. 已发起任务
2. 正在打开页面
3. 正在搜索人群包
4. 正在精准匹配
5. 已匹配成功
6. 正在等待入口出现
7. 已进入透视
8. 正在获取标签数据
9. 已完成
10. 执行失败

## 六、结果需求

用户希望最终结果能回到我们的系统中，而不是只留在官方页面里。

第一版结果需求包括：

1. 能知道任务是否成功。
2. 能看到任务执行到哪一步。
3. 能看到标签透视结果。

后续结果能力可以继续扩展为：

1. 结果预览
2. 结果下载
3. 历史结果复查
4. 结果沉淀和复用

## 七、当前开发阶段要求

当前阶段的重点不是一次性做完所有能力，而是确保流程逐段可测、可见、可验证。

优先级应为：

1. 页面能发起任务。
2. 页面能显示任务状态。
3. 数据引擎阶段可单独测试。
4. 达摩盘阶段可单独测试。
5. 达摩盘透视流程可继续推进。
6. 最终结果可回传。

## 九、最终期望

最终，用户希望这个系统成为一个完整的前端工作台：

1. 用户在我们系统里输入人群包名称。
2. 用户在我们系统里选择标签。
3. 用户点击开始。
4. 系统自动完成页面切换和处理。
5. 用户在我们系统里看到过程。
6. 用户在我们系统里拿到结果。

整个过程尽量减少人工干预，且每一步都应有明确反馈，让用户始终知道系统当前在做什么。
